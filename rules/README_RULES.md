# KURGIN Admin — Rules Checkpoint 01

`kurgin-admin` — отдельная внутренняя админка камней KURGIN.

Это не публичный сайт, не маркетплейс, не CRM, не приложение и не публичный KURGIN Index.

Главная цель первой версии:

```text
загрузка готового Excel KURGIN Score Result
→ поставка
→ камни
→ редактирование
→ цены
→ публикация
→ экспорт
```

## Язык интерфейса

Интерфейс админки по умолчанию должен быть на русском языке.

На русском должны быть страницы, кнопки, подсказки, предупреждения, сообщения ошибок, подтверждения, предпросмотры импорта/публикации и отображаемые статусы.

Английские технические ключи можно хранить внутри данных и кода: `status`, `availability_status`, `stone_id`, `main`, `large`, `published_price`.

В интерфейсе показывать понятные русские названия: «Черновик», «Опубликован», «В наличии», «Продан», «Основной каталог», «Крупные камни», «Публичная цена».


## Компактность интерфейса

Интерфейс админки должен быть максимально компактным.

Правила:

```text
плотные формы
минимум пустого пространства
компактные таблицы
короткие подписи
аккуратные блоки
без лишних больших отступов
```

Это особенно важно для страниц:

```text
Загрузка поставки
Предпросмотр импорта
Камни этой поставки
Таблица камней
Поставки
Цены
Публикация
Экспорт
```

Компактность не должна ухудшать понятность. Если действие опасное, оно всё равно должно иметь ясное предупреждение, предпросмотр и подтверждение.

## Главные правила

1. KURGIN Score в админке не пересчитывается, а импортируется из Excel.
2. ROUND-камни могут публиковаться с KURGIN Score.
3. Не ROUND-камни могут публиковаться без KURGIN Score.
4. `status` и `availability_status` не смешиваются.
5. Импорт не равен публикации.
6. До подтверждения импорта постоянные файлы не меняются.
7. После отмены импорта не остаётся временных файлов, raw-папок или частичных CSV.
8. Каждая загрузка Excel = отдельная поставка.
9. `import_id = shipment_id` в первой версии.
10. Проект должен быть переносимым: не завязывать бизнес-логику на GitHub или Streamlit Cloud.

## Изменение правил

Если новое уточнение меняет правила, сначала нужно предупредить: «Это меняет README_RULES.md».

Если меняется карта этапов, нужно предупредить: «Это меняет карту этапов первой версии».

Порядок: предупреждение → проверка противоречий → обсуждение → принятие пользователем → обновление rules-файлов.


## Удаление поставки на Этапе 1

Этап 1 включает минимальную безопасную возможность удалить конкретную загруженную поставку полностью.

Удаление выполняется только через:

```text
предпросмотр → ввод номера поставки → backup → удаление
```

Удаляется:

```text
камни этой поставки из stones_master.csv
строка поставки из shipments.csv
строка импорта из import_log.csv
папка data/raw/{import_id}/
```

Этап 9 остаётся для сложной архивной очистки, массового удаления и исторических удалений.
